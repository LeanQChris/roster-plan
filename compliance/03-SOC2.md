# SOC 2 Compliance Plan

## 1. Overview
SOC 2 (Service Organization Control 2) is an auditing framework developed by the American Institute of CPAs (AICPA). It evaluates controls based on five Trust Service Criteria.

For v1, we target **SOC 2 Type I** (design of controls at a point in time), with a path to **Type II** (operating effectiveness over 6+ months).

## 2. System Description
The roster application is a multi-tenant, web-based SaaS platform for employee scheduling, time tracking, and workforce management. The system boundaries include:

- **Infrastructure**: Cloud-hosted (AWS/GCP/Azure) across multiple regions. Compute, database, cache, and storage are fully managed cloud services.
- **Software**: A web application with a React frontend and Node.js/Python backend, PostgreSQL database, Redis cache, and object storage.
- **Data**: Employee profiles, shift schedules, clock-in/out records, team structures, company configurations, and audit logs.
- **People**: Platform super admins (infrastructure), company admins (tenant configuration), managers (scheduling), and employees (self-service).
- **Sub-services**: Email delivery (SendGrid/SES), cloud infrastructure provider, monitoring/alerting services.

## 3. Trust Service Criteria Coverage

Each control has a unique ID for traceability to audit evidence.

### 3.1 Security (Common Criteria) — CC
The system is protected against unauthorized access (physical and logical).

| Control ID | Control | Implementation | Evidence |
|---|---|---|---|
| CC-1 | Access control policy | RBAC with least privilege (see RBAC matrix) | Role definitions, permission matrix, user-role assignments |
| CC-2 | Multi-tenant isolation | RLS (Row-Level Security) at database level per company | RLS policies, test proving cross-tenant isolation |
| CC-3 | Authentication | Email + bcrypt password, session tokens, rate limiting | Auth logs, rate limit config, password policy |
| CC-4 | Network security | TLS 1.3 in transit, VPC, security groups | TLS config, network diagram, security group rules |
| CC-5 | Logical access reviews | Quarterly review of all admin accounts | Review meeting notes, sign-off records |
| CC-6 | Termination procedures | Immediate session invalidation on account deactivation | Termination checklists, session revocation logs |
| CC-7 | Intrusion detection | Cloud WAF + anomaly detection on audit log | WAF config, alert triggers, incident response records |
| CC-8 | Vulnerability management | Automated dependency scanning (Dependabot/Snyk), quarterly pen tests | Scan reports, remediation tickets, pen test reports |
| CC-9 | Physical security | Delegated to cloud provider (SOC 2 Type II reports available) | Cloud provider SOC 2 report, sub-processor assessment |
| CC-10 | Environmental safeguards | Delegated to cloud provider (multi-AZ, redundant power/cooling) | Cloud provider SOC 2 report |

## 4. Risk Assessment Methodology
- **Framework**: NIST SP 800-30 Rev. 1 (Guide for Conducting Risk Assessments).
- **Frequency**: Annual full assessment; ad-hoc for significant changes.
- **Process**:
  1. Identify assets and their value (confidentiality, integrity, availability).
  2. Identify threats and vulnerabilities (OWASP Top 10, cloud-specific threats).
  3. Determine likelihood and impact (5x5 matrix).
  4. Calculate risk level (Low / Medium / High / Critical).
  5. Identify and prioritize mitigation controls.
  6. Document residual risk and accept or remediate.
- **Risk register**: Maintained in central document, reviewed quarterly by security team.
- **Acceptance**: Risk acceptance requires sign-off from security lead and CTO.

## 5. Vendor / Sub-Processor Management
- **Risk assessment**: All vendors handling data undergo pre-engagement assessment (security questionnaire, SOC 2 reports, data handling practices).
- **Tiering**:
  - **Tier 1** (critical data access): Cloud provider, email provider, database provider → annual review + contract with security provisions.
  - **Tier 2** (indirect access): Monitoring, logging, CDN → initial assessment + triennial review.
  - **Tier 3** (no data access): Billing, HR tools → standard vendor terms.
- **Contracts**: All Tier 1 and Tier 2 vendors have contractual provisions for: confidentiality, security measures, breach notification, right to audit, data deletion on termination.
- **Ongoing monitoring**: Annual vendor review, security incident notifications from vendors, SLA compliance tracking.
- **Sub-processor list**: Maintained and disclosed to tenant companies.

## 6. System Monitoring & Logging

### 6.1 Audit Log (Immutable Chain)
- Every state change is recorded in `audit_entries` with HMAC chain.
- Logs include: actor, action, resource, old/new values, IP, user agent, timestamp.
- Logs are append-only (triggers prevent UPDATE/DELETE).
- Tampering is detectable by recomputing the hash chain.
- **Integrity verification**: Hash chain recomputed daily, alert on mismatch.

### 6.2 Monitoring
- Application performance monitoring (APM): request latency, error rates, throughput.
- Security monitoring: failed login attempts, suspicious IP patterns, unusual data access.
- Infrastructure monitoring: CPU, memory, disk, network.
- **Control monitoring**: Ongoing testing of key controls (automated where possible), supplemented by periodic manual testing.

## 7. Change Management
All changes to production follow a documented process:
- **Code changes**: Pull request → peer review → CI/CD (lint, test, build, security scan) → deploy.
- **Infrastructure changes**: IaC with version control and approval.
- **Emergency changes**: Hotfix process with mandatory post-deploy review.
- **All changes logged**: Author, reviewer, description, date, rollback plan.
